# Design: Local MCPs appear doubled after enabling

## Architecture

All relevant code is in `helix/frontend/src/components/app/Skills.tsx`.

### Current Flow (Buggy)

The `allSkills` list is assembled from multiple sources:

```
allSkills = [
  ...BASE_SKILLS,        // includes hardcoded "Drone CI" (id: drone-ci) and "GitHub" (id: github-mcp)
  ...customApiSkills,    // from app.apiTools
  ...backendSkills,      // from backend tool registry
  ...mcpSkills,          // from app.mcpTools where transport is HTTP/SSE (not stdio)
  CUSTOM_API_SKILL,
  CUSTOM_MCP_SKILL,
  ...localMcpSkills,     // from app.mcpTools where transport === 'stdio' || mcp.command
  CUSTOM_LOCAL_MCP_SKILL,
]
```

The problem: "Drone CI" and "GitHub" exist in `BASE_SKILLS` as hardcoded tiles AND get dynamically generated by `localMcpSkills` once enabled (because enabling adds them to `app.mcpTools` with `transport: 'stdio'`).

### Fix Strategy

**Exclude hardcoded names from `localMcpSkills` generation.**

In the `localMcpSkills` useMemo, filter out any MCP entries whose name matches a skill already in `BASE_SKILLS`. The hardcoded `BASE_SKILLS` entries for Drone CI and GitHub already have dedicated dialog components (`DroneCiSkill`, `GitHubMcpSkill`) and dedicated `isSkillEnabled` checks, so the dynamic tiles are purely redundant.

### Concrete Change

In `Skills.tsx`, the `localMcpSkills` useMemo (~line 658) currently does:

```ts
return app.mcpTools
  .filter(mcp => mcp.transport === 'stdio' || mcp.command)
  .map(mcp => ({ ... }));
```

Change to:

```ts
// Names of skills already defined in BASE_SKILLS — don't generate dynamic duplicates
const baseSkillNames = new Set(BASE_SKILLS.map(s => s.name));

return app.mcpTools
  .filter(mcp => mcp.transport === 'stdio' || mcp.command)
  .filter(mcp => !baseSkillNames.has(mcp.name || ''))
  .map(mcp => ({ ... }));
```

This is a simple, safe exclusion: if a hardcoded skill tile already exists for a given name, don't create a second dynamic one.

## Alternatives Considered

1. **Remove Drone CI and GitHub from BASE_SKILLS and rely solely on dynamic tiles.** Rejected — the hardcoded entries provide better UX (custom icons, descriptions, dedicated config dialogs) even when the MCP isn't yet enabled. Dynamic tiles only appear *after* enabling.

2. **Deduplicate in `allSkills` instead of in `localMcpSkills`.** This would work but is less targeted. Filtering at the source (`localMcpSkills`) is simpler and makes the intent clear.

## Scope

- **1 file changed**: `frontend/src/components/app/Skills.tsx`
- **~3 lines added** (the `baseSkillNames` set and the extra `.filter()`)
- No backend changes, no new dependencies, no API changes

## Risks

- If a user creates a custom local MCP that happens to share a name with a BASE_SKILLS entry (e.g., names their custom MCP "GitHub"), the custom tile would be suppressed. This is an unlikely edge case and arguably correct behavior — the hardcoded tile would handle it via its dedicated dialog.
