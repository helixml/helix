- [x] **Expose :443 (and :80) on deployment surfaces.** This is a gap from 002096: the existing HTTP-01 `auto` mode binds `:443` + `:80` inside the API container but no compose file or Helm Service ever mapped a host/Service port to them, so auto-mode was operationally dead. Fixed:
  - `docker-compose.tls.yaml` (new): opt-in compose overlay. Host ports configurable via `.env` (`VHOST_HTTPS_PORT`, `VHOST_HTTP_PORT`).
  - `stack` (script): auto-detects `HELIX_VHOST_TLS_MODE=auto` in `.env` or the shell and threads `-f docker-compose.tls.yaml` through the mutating compose calls (`up -d`, `down`, `build`, `rebuild`). Operators don't pass any extra flag — just set the env var.
  - `install.sh`: same auto-detection in the controlplane install/reinstall/upgrade paths via a new `helix_compose_args` helper. Three new flags write the TLS env vars into `.env` at install time: `--vhost-tls-mode`, `--letsencrypt-email`, `--cloudflare-api-token` (last one implies `--vhost-tls-mode=auto`). Both the full controlplane install path AND `do_upgrade` (the `--upgrade` lightweight path) fetch `docker-compose.tls.yaml` from the GitHub release — `do_upgrade` does so only when missing, so upgrades from pre-overlay releases work end-to-end without forcing a full re-install.
  - `.drone.yml`: release pipeline now uploads `docker-compose.tls.yaml` as a release asset.
  - Helm chart values: add `controlplane.vhostTLS.enabled` (default `false`), `controlplane.vhostTLS.httpsPort` (default `443`), `controlplane.vhostTLS.httpPort` (default `80`).
  - Helm chart Service: when `vhostTLS.enabled`, add `https` and `http-acme` ports targeting the named container ports.
  - Helm chart Deployment: when `vhostTLS.enabled`, declare matching `containerPort: 443` and `containerPort: 80`.
