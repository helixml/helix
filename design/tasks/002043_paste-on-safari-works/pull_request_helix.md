# Fix Safari copy from remote desktop to system clipboard

## Summary

On Safari, pressing Cmd+C inside the remote-desktop stream viewer was
showing a green "Copied" toast but never actually writing to the macOS
clipboard. Root cause: the existing handler used `setTimeout(300ms)` ‚Üí
fetch ‚Üí `navigator.clipboard.writeText()`, and WebKit/Safari enforces
the rule that a clipboard write must be triggered *synchronously inside
the originating user-gesture task*. After the 300ms timeout, the gesture
context is gone and Safari silently rejects the write. The catch block
still fired the success toast.

Fixed by:

1. **Gesture-anchored clipboard write via `ClipboardItem({ promise })`**
   ‚Äî synchronously start `navigator.clipboard.write()` inside the
   Cmd+C keydown handler with a `ClipboardItem` whose Promise<Blob>
   values resolve later. This is the WebKit-blessed pattern from
   [the 2020 Async Clipboard API blog post](https://webkit.org/blog/10855/async-clipboard-api/).
2. **Bounded adaptive polling** in place of the hard-coded 300ms wait
   ‚Äî snapshot the clipboard hash in parallel with forwarding Ctrl+C,
   then poll every ~30ms for up to ~500ms, resolve the moment the
   hash changes. Typical resolution: 30‚Äì90ms.
3. **Dual-MIME ClipboardItem** carries both `text/plain` and `image/png`
   so images on the remote clipboard also round-trip to local on
   Safari. Paste destinations pick the MIME they prefer. (Trade-off:
   pasting a text-copy into an image-only app yields a 0-byte image ‚Äî
   user retries with a text destination.)
4. **Extended macOS Wails postMessage bridge to carry images** via a
   new cgo binding to `NSPasteboard` (`for-mac/clipboard_darwin.go`).
   `SetClipboardImagePNG` writes `NSPasteboardTypePNG`;
   `GetClipboardImagePNG` reads `NSPasteboardTypePNG` and falls back
   to `NSPasteboardTypeTIFF` (transcoding to PNG) so screenshots taken
   via `Cmd+Shift+Ctrl+4` paste correctly.
5. **Removed the 2.7-second background clipboard polling `useEffect`**:
   it couldn't work on Safari (no user gesture ‚Üí silent fail) and it
   raced with explicit Cmd+C on slow networks. Cmd+C is now the only
   path remote‚Üílocal. Accepted regression: copying inside the remote
   via right-click no longer auto-populates local clipboard.
6. **Truthful toasts** ‚Äî green "Copied text" / "Copied image" only on
   successful local write; error variant ("Copied on remote ‚Äî local
   clipboard blocked") when the local write is rejected.

## Changes

- `frontend/src/components/external-agent/DesktopStreamViewer.tsx`:
  - New `clipboardWrite` / `clipboardReadAny` helpers with a mime
    discriminator, replacing `clipboardWriteText` / `clipboardReadText`.
  - Rewrote the Cmd+C branch in `handleKeyDown` (gesture-anchored
    `ClipboardItem.write` with bounded poll).
  - Rewrote the Cmd+V branch to use the unified `clipboardReadAny`
    which carries images through the iframe bridge.
  - Extended the native `handlePaste` DOM event to detect
    `clipboardData.files` for synchronous image paste.
  - Deleted: the 2.7s auto-sync `useEffect`, `lastRemoteClipboardHash`
    ref, `lastAutoSyncedText` ref, and the paste-flow short-circuit
    that depended on the latter.
- `for-mac/clipboard_darwin.go` (new): cgo binding to NSPasteboard with
  `SetClipboardImagePNG` / `GetClipboardImagePNG` (+ TIFF fallback).
- `for-mac/frontend/src/App.tsx`: extended `handleMessage` to route
  `mime: "image/png"` write/read messages to the new Go methods; kept
  acceptance of old `{ type, text }` writes for back-compat.
- `for-mac/frontend/wailsjs/go/main/App.{d.ts,js}`: hand-added bindings
  for the two new methods (normally regenerated by `wails dev`).
