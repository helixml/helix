## Merge Strategy

Use `git merge upstream/main` (consistent with every prior merge — preserves
Helix commit history and keeps conflict resolution traceable). Do **not** rebase.

```bash
cd /home/retro/work/zed

```

### Conflict-resolution principles (carried from 001909 / 001980 / 001996 / 002029 / 002077 / 002100)

1. **Preserve Helix behaviour by default.** Never let cfg-gated Helix code leak into unconditional paths.
2. **Retire, don't silently drop.** If upstream has absorbed a Helix patch (e.g. a fix now shipped upstream), drop the Helix-side copy and write an explicit retirement note in `portingguide.md`. A deliberate removal is fine; a silent one is not.
3. **Prefer upstream ordering** for shared lists (model lists, feature lists, match arms) — smaller diff, less future conflict.
4. **`Cargo.lock`**: always `git checkout --theirs` (upstream); regenerated by the next build.
5. **`Cargo.toml`**: take upstream's `agent-client-protocol = "=1.1.0"` pin. Keep Helix workspace members (`external_websocket_sync`, `cloud_api_types`) and `rust-embed`'s `debug-embed` feature.
6. **CI / `.github/workflows/`**: always accept upstream — Helix doesn't use Zed's CI.
7. **Upstream renames/deletes**: after the merge — even for cleanly auto-merged files — grep for old identifiers in cfg-gated regions. Two `agent_configuration/*_modal.rs` files are deleted upstream this window.
8. **Trait-signature changes**: resolve compile-driven; the post-merge build is the safety net. This window that includes the whole ACP `1.1.0` API surface.
9. **Document every non-trivial resolution in `portingguide.md` immediately.** Given 409 commits, incremental notes are not optional.
10. **If a conflict is too risky** (most likely an ACP-`1.1.0` trait change with unclear semantics): stop, document, and escalate to the user rather than guess.
