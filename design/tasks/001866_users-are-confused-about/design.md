# Design: Claude Code Token Validation

## Token Format Analysis

From real examples and codebase analysis:

| Token Type | Prefix | Example |
|-----------|--------|---------|
| Claude Code setup token | `sk-ant-oat` | `sk-ant-oat59jYxX8OG76RMOi-...` |
| Anthropic API key | `sk-ant-api03-` | `sk-ant-api03--_4V9f-bhQsZTyc...` |
| OAuth refresh token | `sk-ant-ort01-` | (internal, not user-facing) |

Setup tokens from `claude setup-token` start with `sk-ant-oat`. This is the only format that should be accepted in the setup token dialog.

## Architecture

Validation happens in two places — frontend (fast feedback) and backend (defense in depth).

### Frontend Validation (ClaudeSubscriptionConnect.tsx)

Add a validation function that runs on every change to `tokenValue`:

```typescript
function validateSetupToken(token: string): string | null {
  const trimmed = token.trim()
  if (!trimmed) return null // empty = no error yet

  if (trimmed.startsWith('sk-ant-api')) {
    return 'This looks like an Anthropic API key, not a Claude Code setup token. Run `claude setup-token` in your terminal to generate the correct token.'
  }

  if (!trimmed.startsWith('sk-ant-oat')) {
    return "This doesn't look like a valid Claude Code setup token. Run `claude setup-token` to generate one."
  }

  if (trimmed.length < 50) {
    return 'This token appears to be incomplete. Make sure you copied the full token.'
  }

  return null // valid
}
```

Wire this into the component:
- Compute `validationError = validateSetupToken(tokenValue)` on each render.
- Show `validationError` as an `<Alert severity="error">` below the text field (same spot as `submitError`).
- Disable the "Connect" button when `validationError` is non-null.
- Keep existing `submitError` handling for backend errors.

### Backend Validation (claude_subscription_handlers.go)

In the `createClaudeSubscription` handler, add validation before encryption:

```go
if createReq.SetupToken != "" {
    token := strings.TrimSpace(createReq.SetupToken)

    if strings.HasPrefix(token, "sk-ant-api") {
        return nil, system.NewHTTPError400(
            "This is an Anthropic API key, not a Claude Code setup token. " +
            "Run 'claude setup-token' in your terminal to generate the correct token.")
    }

    if !strings.HasPrefix(token, "sk-ant-oat") {
        return nil, system.NewHTTPError400(
            "Invalid setup token format. " +
            "Run 'claude setup-token' to generate a valid token.")
    }

    // ... existing encryption logic
}
```

### UI Label Changes (ClaudeSubscriptionConnect.tsx)

- Text field `label`: `"Your Token"` -> `"Claude Code Setup Token"`
- Text field `placeholder`: `"Paste your token here..."` -> `"Paste your Claude Code setup token here..."`

## Decisions

**Why validate on both frontend and backend?** Frontend gives instant feedback. Backend is the safety net — other clients or direct API calls must also be rejected. This is a system boundary (user input), so validation belongs here.

**Why prefix-match `sk-ant-oat` specifically?** This is the known prefix for setup tokens generated by `claude setup-token`. If Anthropic changes the prefix in the future, we update the validation. The alternative — accepting anything — is the current broken behavior.

**Why not validate the token against Anthropic's API?** A setup token is an opaque credential — there's no lightweight "verify this token" endpoint. Format validation catches the common mistake (wrong credential type) without adding latency or external dependencies.

## Codebase Notes

- Frontend component: `frontend/src/components/account/ClaudeSubscriptionConnect.tsx`
- Backend handler: `api/pkg/server/claude_subscription_handlers.go`, lines 67-81
- The `handleSubmitToken` function (line 115) uses raw `api.post` — the CLAUDE.md says to use the generated API client, but this is a pre-existing pattern in this file. Changing to the generated client is out of scope for this task.
- Token prefixes: `sk-ant-oat` = setup token, `sk-ant-api03-` = API key. Backend check uses `sk-ant-api` (without version suffix) to catch all API key versions.

## Implementation Notes

- Frontend validation function `validateSetupToken()` added as a module-level function (not inside the component) since it's pure and stateless.
- Validation result is computed on every render via `const tokenValidationError = validateSetupToken(tokenValue)` — no need for useEffect/useMemo since it's a cheap string check.
- Backend validation added at the top of the setup token branch (before marshalling/encryption), so invalid tokens never touch the crypto layer.
- The backend reuses the existing `strings.TrimSpace` that was already applied to `createReq.SetupToken` — we introduced a `token` local variable to avoid trimming twice.
- OAuth flow is completely unaffected — it's a separate `else` branch that doesn't check `SetupToken` at all.
- The `fill` MCP tool sets DOM values without triggering React's synthetic onChange, so manual testing required `type_text` instead. Real users typing or pasting will trigger onChange correctly.
- Frontend build has a pre-existing permissions issue with `frontend/dist/` (bind mount), but TypeScript type-checking (`tsc --noEmit`) passes cleanly.
