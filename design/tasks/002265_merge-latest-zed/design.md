## Merge Strategy

`git merge upstream/main` — never rebase. Preserves Helix commit history and keeps
conflict resolution traceable. Consistent with every prior merge in the series.

```bash
cd /home/retro/work/zed
git remote add upstream https://github.com/zed-industries/zed.git   # if missing
git fetch upstream --no-tags && git fetch origin
git checkout main && git pull origin main
git checkout -b feature/002265-merge-latest-zed
# START the portingguide.md ## Merge 002265 entry NOW (before resolving)
git merge upstream/main
# Resolve conflicts one at a time; append a portingguide bullet as EACH is resolved.
# Then: take upstream ACP pin → build → critical-fix grep → unit tests → E2E.
git push -u origin feature/002265-merge-latest-zed
# Write pull_request_zed.md + pull_request_helix.md; bump ZED_COMMIT in helix; push.
```

### Conflict-resolution principles (carried from the whole series)

1. **Preserve Helix behaviour by default.** Never let cfg-gated Helix code leak
   into an unconditional path. Built-in agent hiding uses `cfg(not(feature = ...))`.
2. **Retire, don't silently drop.** If upstream absorbed a Helix patch, drop the
   Helix copy and write an explicit retirement note. Deliberate removal is fine;
   a silent one is not.
3. **Prefer upstream ordering** for shared additive lists (model lists in
   `anthropic/src/anthropic.rs`, feature lists, match arms) — smaller diff.
4. **`Cargo.lock`**: `git checkout --theirs`; regenerated by the next build.
5. **`Cargo.toml`**: take upstream's `agent-client-protocol` pin. Keep Helix
   workspace members (`external_websocket_sync`, `cloud_api_types`) and
   `rust-embed`'s `debug-embed` feature.
6. **`.github/workflows/` / Zed CI**: accept upstream — Helix doesn't use it. But
   do **not** disturb `.drone.yml`'s `cargo build --locked` (Helix CI needs it).
7. **Upstream renames/deletes**: after the merge — even for cleanly auto-merged
   files — grep for old identifiers in cfg-gated regions. *Auto-merged ≠ correct.*
8. **Trait-signature / `non_exhaustive` changes**: resolve compile-driven; the
   post-merge build is the safety net. Convert struct literals to builders; check
   `ErrorCode` arms if ACP bumped.
9. **Document every non-trivial resolution in `portingguide.md` immediately.**
10. **If a conflict is too risky** (most likely an ACP trait change with unclear
    semantics, or upstream's session/resume UI colliding with `from_existing_thread`):
    stop, document, and escalate to the user rather than guess.

## Key Decisions & Load-bearing Patterns

- **GPUI subscription gotcha (brief #6):** `subscribe_in` is silently dropped when
  no window context exists. Keep the **channel-based** pattern for UI updates; use
  `App::subscribe` (windowless) for WebSocket forwarding; keep windowless
  `cx.subscribe()` in `thread_service.rs` for incremental streaming.
- **`AcpThreadEvent::Stopped(StopReason)`** is a tuple variant — matches need
  `Stopped(_)`, test code included. If ACP reshapes this, re-verify Fix #6/#9.
- **`BaseView::*` / `ContextServerStatus::*` arms** must stay exhaustive in
  Helix-mode code (UI state queries + headless responder). Add any new upstream
  variant.
- **`HeadlessConnection` is dead code** (since 001909) — ignore stale guide refs.
- **`from_existing_thread()` field set** is build-gated — a green build confirms no
  drift; still diff it field-by-field against its `::new` sibling.
- **`title_bar`'s `external_websocket_sync` dep MUST be `optional = true`** with the
  matching `[features]` entry.
- **`go mod tidy` in `e2e-test/helix-ws-test-server/`** before E2E — the runner
  doesn't tidy itself. Never `--no-build` when investigating E2E failures.
- **CI `--locked` (brief #5):** `.drone.yml`'s `cargo build --locked` must survive —
  the Zed source is mounted read-only in CI, so an unlocked build (which rewrites
  `Cargo.lock`) fails. Verify after resolving `Cargo.lock`.

```bash

```
The task brief's raw invocation (`docker build … -f …/Dockerfile . && docker run … zed-ws-e2e`)
targets the same test; `run_docker_e2e.sh` is the established wrapper. **Enumerate
the actual phase list at runtime** — do not assume a count. All phases must pass
for both agents, including the brief's 4 named phases and UI-state queries. One
retry permitted per agent for the Claude Phase-1 npm flake / single API-latency hiccup.
If any phase fails: do **not** complete — diagnose, fix, document, re-run.

## Documentation Updates (incremental — not retrospective)

New `## Merge 002265 (2026-07-20)` section at the top of `portingguide.md`'s
merge-history list, started when `git merge` is issued:

1. **Window summary** — measured commit count + window + upstream HEAD SHA + fence + ACP state.
2. **`### Conflicts and Resolutions`** — one subsection per manual conflict (file/region,
   upstream change, what was kept from each side and why). Or explicit "0 conflicts".
3. **`### ACP bump — repaired sites`** — one bullet per fixed call site, or "no ACP bump".
4. **`### Retired Helix patches`** — with justification, or explicit "none".
5. **Helix-surface survival check** — per-area confirmation (Fix 1b position; 3× `// HELIX:`
   markers; Critical Fixes #1/#3/#6/#8/#9/#11; recent PR surface intact).
6. **Commit-history table** extended. New Rebase-Checklist entry only if a genuinely new
   recurring gotcha is confirmed. Correct/delete any stale entries found.

## Out of Scope

See requirements. In short: no net-new Helix features, no E2E-assertion changes
(unless an upstream ACP API change forces it), no upstreaming Helix patches, no
Helix-crate refactors beyond what the merge forces, no adoption of upstream
slash-command/elicitation/sandboxing UX into Helix-mode flows beyond compiling.
