# Requirements: Generate Snappy LLM Titles for Spec Tasks

## Background

When a user creates a spec task, the task's display name is generated by
`GenerateTaskNameFromPrompt()` (see
`api/pkg/services/spec_driven_task_service.go:1691`). This function just
collapses whitespace in the original prompt, takes the first ~60 characters,
and appends `...` if truncated. So a prompt like:

> "Give spec tasks a generated title, and show this on the UI. Right now it
>  defaults to the first line, make it a snappy title instead."

becomes the tab/card title:

> "Give spec tasks a generated title, and show this on the UI...."

That's noisy and hard to scan in the Kanban board and task list. The user
wants a **snappy, LLM-generated title** to appear *as soon as the task is
created* — not only after the agent has finished spec planning and pushed
`requirements.md` (the current point at which `task.Name` gets replaced with
the parsed H1 — see `api/pkg/services/git_http_server.go:1660-1671`).

The `SpecTask` schema already has a `short_title` column intended for this
exact purpose (`api/pkg/types/simple_spec_task.go:121`), but no code path
populates it today. The frontend already prefers
`user_short_title || short_title || name` in `TabsView.tsx`, so populating
`short_title` cleanly slots into the existing display chain.

## User stories

### Story 1 — Snappy title on the Kanban/tab/card view

**As a** user creating a new spec task,
**I want** the task to display a short, descriptive title (e.g. "Snappy
spec-task titles") instead of the truncated raw prompt,
**so that** I can quickly scan my Kanban board, tab strip and task list.

Acceptance criteria:

- WHEN a task is created via `POST /v1/spec-tasks` or the Just Do It path,
  THEN within ~5 seconds the task row in the DB SHALL have `short_title` set
  to an LLM-generated string of ≤ 60 characters that summarises the original
  prompt.
- WHEN `short_title` is set, THEN the tab strip (`TabsView.tsx`), the task
  card (`TaskCard.tsx`), the task table (`TasksTable.tsx`), the Kanban board
  (`SpecTaskKanbanBoard.tsx`), the empty-state list (`EmptyTasksState.tsx`)
  and the design-review dialog (`SpecTaskDetailContent.tsx:910`) SHALL all
  show `user_short_title || short_title || name`, preferring the shortest
  human-friendly value available.
- WHEN the user hovers/tooltips a task, THEN the original prompt SHALL still
  be reachable (as it already is via `task.description` in the existing
  tooltip path).

### Story 2 — Graceful fallback when LLM is unavailable

**As an** operator running Helix without a configured enrichment model,
**I want** task creation to still succeed and show a sensible title,
**so that** the feature degrades gracefully.

Acceptance criteria:

- IF the system-settings enrichment model
  (`settings.KoditEnrichmentProvider` / `settings.KoditEnrichmentModel`) is
  not configured OR the LLM call returns an error / timeout,
  THEN the task SHALL fall back to the existing `Name` behaviour (first line
  of the prompt, capped at 60 chars) and `short_title` SHALL remain empty.
- IF the LLM call eventually succeeds on retry, `short_title` MAY be filled
  in later — but task creation MUST NOT block on it.

### Story 3 — Agent-generated H1 still wins

**As a** user who lets the agent run spec planning,
**I want** the snappy title the agent chose in `requirements.md` to still
appear once planning completes,
**so that** the title reflects the agreed design, not just my initial prompt.

Acceptance criteria:

- WHEN the planning agent pushes `requirements.md` with a `# Requirements:
  <Descriptive Title>` H1, THEN `SpecTitleFromRequirements` SHALL continue to
  update `task.Name` as it does today (see
  `git_http_server.go:1660-1671`).
- The same code path SHALL also update `task.ShortTitle` with the parsed H1
  (so the H1 from the agent supersedes the LLM-generated draft from
  step 1).

### Story 4 — User override is preserved

**As a** user who has manually renamed a tab (double-click → edit),
**I want** my override to keep showing,
**so that** auto-generation never clobbers my choice.

Acceptance criteria:

- WHEN `user_short_title` is non-empty, the display chain
  `user_short_title || short_title || name` already prefers it; this
  ordering MUST NOT change.
- The task-rename handler (`spec_driven_task_handlers.go:1077`) MUST
  continue to set ONLY `user_short_title` and never touch `short_title` /
  `name`.

### Story 5 — Cost & rate-limit discipline

**As an** operator,
**I want** title generation to use the same cheap "small" enrichment model
that summarises sessions today,
**so that** I'm not paying frontier-model prices for a one-line title.

Acceptance criteria:

- The implementation SHALL reuse the existing `SummaryService` pattern
  (`api/pkg/server/summary_service.go`), in particular:
  - the `KoditEnrichmentProvider` / `KoditEnrichmentModel` system settings,
  - the `maxConcurrent` rate limit (5),
  - the 30-second per-call timeout,
  - `MaxTokens: 50`, `Temperature: 0.3`,
  - the trim-quotes + truncate-at-word-boundary cleanup.
- No new system-settings keys SHALL be introduced for this feature.
